# CP-Helper — комплексный анализ проекта

*Независимый аудит: архитектура, сильные и слабые стороны, безопасность, предложения по развитию.*
*Дата: 16.07.2026. Проанализировано ~2 830 строк JS/JSX + Rust-обёртка, правила Firestore, CI, конфигурация Tauri.*

---

## 1. Что это за проект

Десктопное приложение (Tauri) для управления **CP** (Constant Party) в Lineage 2 для сквада «0utLaw / UASQUAD». Закрывает пять задач: ростер пачки, казна (adena / Master Coins), чек-листы квестов по классу и расе, общие задачи (prime/offprime) и отчёт дисциплины (кто сдал квесты и минимум adena за день). Есть система ролей (GUEST → MEMBER → OFFICER → PL), русский и украинский языки, авто-обновление через GitHub Releases.

**Стек:** Tauri 2 (Rust — только окно, трей, updater, dialog; своих команд нет) + React 19 + Vite 8 + react-router-dom 7 (HashRouter). Бэкенд — Firebase Auth (email/пароль) и Firestore с realtime-подписками. Вся бизнес-логика во фронтенде, напрямую работает с Firestore; безопасность держится **исключительно** на `firestore.rules`.

## 2. Сильные стороны

**Архитектура frontend аккуратная и последовательная.** Слой сервисов (`src/services/*`) чисто отделён от компонентов: каждый модуль инкапсулирует работу с одной коллекцией и отдаёт либо подписку (`subscribeToX`), либо мутацию. Компоненты не знают о структуре Firestore. Это редкая для пет-проектов дисциплина, и она сильно упростит развитие.

**Realtime «из коробки».** Все экраны живут на `onSnapshot` — данные обновляются у всех участников мгновенно без ручного рефреша. Для инструмента координации пачки это правильный выбор.

**Ролевая модель продумана концептуально.** Четыре роли с наследованием прав, разграничение UI по ролям, самоповышение роли заблокировано и на клиенте, и в правилах (`diff().affectedKeys().hasAny(['role'])`).

**Безопасность Tauri настроена грамотно.** CSP не разрешает `unsafe-eval`, `script-src 'self'`, `connect-src` ограничен доменами Firebase/Google. Updater подписан minisign, приватный ключ — только в GitHub Secrets, публичный — в конфиге. Capabilities минимальны (`core:default` + `updater:default`). Это заметно лучше, чем у типичного Tauri-приложения.

**CI с защитой от рассинхрона версий.** Workflow перед сборкой сверяет git-тег, `package.json` и `tauri.conf.json` и падает при расхождении — хорошая инженерная привычка.

**Есть ErrorBoundary и обработка ошибки авторизации** с осмысленным fallback-экраном и кнопкой перезагрузки.

**i18n вынесен централизованно** (`i18n/translations.js`, `LanguageContext`), в компонентах нет захардкоженных строк — переключение языка и добавление новых локалей дёшевы.

**Оптимистичный код читаемый:** понятные имена, небольшие функции, отсутствие «магии». Линтер (oxlint) подключён.

## 3. Слабые стороны и технический долг

**Вся логика и валидация — на клиенте.** Firestore-правила проверяют только роль автора записи, но **не содержимое**. Офицер (или кто угодно с валидным токеном через прямой вызов SDK) может записать транзакцию с любым `type`, `currency`, отрицательным или строковым `amount`, произвольным `member`/`addedBy`. Нет server-side валидации схемы вообще. Для «казны», где считаются реальные игровые ценности, это главный структурный риск.

**Модель данных квестов и лога не масштабируется.** `config/questData`, `config/questLog`, `config/minContributions` — это single-document хранилища. `questLog` хранит прогресс ВСЕХ игроков по ВСЕМ квестам в одном документе `config/questLog` в виде вложенной мапы `{userId: {questName: bool}}`. Лимит документа Firestore — 1 МБ; при росте пачки и числа квестов документ пухнет, а каждая отметка квеста переписывает весь merge-слой. Правильнее — подколлекция `questLog/{userId}/quests/{questId}` или отдельные документы на игрока.

**Расчёт баланса казны — на клиенте, полным перечитыванием.** `subscribeToTransactions` тянет ВСЕ транзакции за всю историю и суммирует их в браузере при каждом изменении. Через год работы это тысячи документов на каждый рендер Dashboard. Нужны агрегаты (Firestore aggregation queries или отдельный документ-агрегат, обновляемый транзакционно).

**Курс обмена adena/MC (`exchangeRate = 29000`) захардкожен и хранится только в локальном стейте** Treasury — не персистится, у каждого пользователя свой, «Итого в adena» у разных людей отличается. Должен жить в `config`.

**Дублирование логики между экранами.** `getClassDetails`, фильтрация активных участников ростера (`m.name !== '—' && m.userId && m.userId !== '__occupied__'`), форматирование чисел `Intl.NumberFormat('ru-RU')`, вычисление прогресса квестов — всё скопировано в Dashboard, MemberProgress, Layout, Treasury. Напрашиваются общие хуки/утилиты (`useActiveRoster`, `formatNumber`, `getQuestProgress`).

**«Магические» строки как данные.** `'__occupied__'`, `'—'` как маркеры пустого/занятого слота, `tag: 'prime'|'offprime'`, `currency: 'adena'|'mc'` разбросаны строками без единого источника истины (enum/константы). Легко ошибиться при опечатке.

**Обработка ошибок непоследовательна.** Часть мутаций обёрнута в try/catch с `alert()`, часть (`toggleTask`, `deleteTask`, `toggleQuestCompletion`, `updateRosterSlot`) вызывается без обработки — при отказе Firestore (в т.ч. из-за правил, см. §4) пользователь не увидит ничего, промис молча отклонится. UX-обратной связи через `alert()`/`window.confirm()` для десктоп-приложения тоже примитивен — стоит перейти на тосты/модалки Tauri.

**Полное отсутствие тестов.** Ни unit, ни интеграционных, ни тестов правил Firestore (а именно правила — единственный слой безопасности; их обязательно нужно покрывать эмулятором `@firebase/rules-unit-testing`).

**Мусор в репозитории:** закоммичены `.DS_Store` (в нескольких папках), `temp.html`, `musthavequests.pdf` (153 КБ, исходник данных), каталог `dist/` и `0utLaw/`. `.gitignore` игнорирует `dist`, но он уже в индексе. Стоит вычистить и добавить `*.DS_Store`, `serviceAccountKey.json`, `.env` в игнор явно.

**`console.error` как основной механизм логирования** в проде. У Tauri-приложения нет открытой консоли — ошибки уходят в никуда. Нужен файловый лог (плагин `tauri-plugin-log` уже подключён, но только в debug).

**Нет офлайн-состояния и индикаторов загрузки** на большинстве экранов — при обрыве связи Firestore realtime молча замирает.

## 4. Безопасность и уязвимости

### 🔴 Критично

**1. Рассинхрон правил и логики: MEMBER физически не может отметить свой квест.**
Dashboard даёт рядовому участнику (MEMBER) отмечать выполнение своих квестов через `toggleQuestCompletion` → запись в `config/questLog`. Но правило для коллекции `config`:
```
match /config/{doc} { allow write: if isOfficer(); }
```
разрешает запись **только офицерам и PL**. Значит, у MEMBER запись отклоняется Firestore, а вызов идёт без try/catch → отметка молча не сохраняется. Это одновременно функциональный баг и симптом того, что модель прав не соответствует UX. Лечится либо переносом questLog в подколлекцию с правилом «пользователь пишет только свой прогресс», либо явным правилом на `config/questLog`, разрешающим участнику писать только свой ключ.

**2. Казна без валидации содержимого — риск порчи данных и накрутки.**
Правило `transactions` проверяет только `isOfficer()` на create, но не структуру. Любой офицер (аккаунт которого может быть скомпрометирован) или самостоятельный вызов SDK может записать некорректную/поддельную транзакцию, отрицательные суммы, чужого `addedBy`. Нет неизменяемости истории (update/delete у PL есть — историю можно переписать без следа). Для учётной системы нужна валидация схемы в правилах + запрет изменения ключевых полей после создания + серверный timestamp (сейчас `serverTimestamp()` есть — хорошо, но не форсится правилом).

### 🟠 Средне

**3. Whitelist регистрации читается публично и без авторизации.**
`config/registration` разрешён к чтению всем (`doc == 'registration'` без `isAuth`). Это необходимо для проверки до логина, но означает, что список `allowedEmails` (приватные email участников) может прочитать любой аноним, знающий проект. Утечка персональных данных. Лучше проверять доступ к регистрации через Cloud Function, а не отдавать whitelist клиенту.

**4. Регистрация GUEST-аккаунтов ничем не ограничена.**
Любой может создать аккаунт (роль GUEST безвредна сама по себе), но нет rate-limit / captcha / verify email — открыто для спама и создания мусорных документов `users`. GUEST по правилам создаётся с `role == 'GUEST'` — это ограничение соблюдено, но объём мусора не лимитирован.

**5. `firestore.indexes.json` пуст, а запросы используют `orderBy`.**
Пока запросы простые (single-field orderBy обслуживаются автоматически), но при добавлении составных фильтров (`where + orderBy`) правила упадут в рантайме без задеплоенного индекса — легко пропустить.

### 🟡 Низко / информационно

**6. Firebase apiKey в `src/firebase.js` — это НЕ уязвимость.** Это публичный идентификатор проекта, он и должен быть в клиенте; безопасность обеспечивают правила и настройки Auth. Отмечаю специально, потому что это частый источник ложной паники. **Но** стоит в Firebase Console включить **App Check** (attestation), ограничить методы входа и настроить **authorized domains**, иначе ключом можно пользоваться из любого стороннего клиента в обход вашего приложения.

**7. `set_pl.js` выдаёт роль PL по email и содержит дефолтный email в коде.** Скрипт админский, работает только с service account, но захардкоженный `a0w.k1m@gmail.com` как дефолт — плохая практика; вынести в обязательный аргумент.

**8. `updateProfile` / `setDoc` при первом входе выполняются в обработчике `onAuthStateChanged` без защиты от гонок** — при двойном срабатывании возможна двойная запись. Малый риск.

**9. Нет проверки силы пароля** при регистрации (полагается только на дефолт Firebase — 6 символов).

Итог по безопасности: периметр Tauri (CSP, updater, capabilities) сделан хорошо; **основной риск — в том, что Firestore-правила проверяют "кто пишет", но не "что пишут", и модель прав не полностью совпадает с UX.**

## 5. Предложения по улучшению (приоритизировано)

### Приоритет 1 — надёжность и безопасность данных
1. **Синхронизировать права questLog с UX** (перенести в подколлекцию `questProgress/{userId}` с правилом «пишу только своё»). Закрывает критичный баг №1.
2. **Добавить валидацию схемы во все правила Firestore** для `transactions`, `roster`, `tasks`, `config`: проверять типы, диапазоны, запрещать смену `addedById`/`timestamp`/`amount` после создания. Закрывает риск №2.
3. **Покрыть правила тестами** через Firebase Emulator + `@firebase/rules-unit-testing` — это самый дешёвый способ не сломать безопасность при доработках.
4. **Обернуть все мутации в единый обработчик ошибок** с тостами, чтобы отказ правил был виден пользователю.

### Приоритет 2 — масштабируемость
5. **Агрегат казны** вместо суммирования всей истории на клиенте: документ `config/treasuryTotals`, обновляемый в транзакции при добавлении записи, либо Firestore aggregation queries. Плюс пагинация истории транзакций.
6. **Перенести `exchangeRate` и другие настройки в `config`** — единый источник для всех.
7. **Разбить single-doc хранилища** questLog/questData под рост данных.

### Приоритет 3 — качество кода
8. **Вынести общие утилиты и хуки:** `useActiveRoster`, `formatNumber`, `getQuestProgress`, константы (`ROLES`, `CURRENCIES`, `SLOT_EMPTY`, `TASK_TAGS`) в единый модуль.
9. **Файловое логирование в проде** (`tauri-plugin-log` вне debug) + опциональная отправка ошибок.
10. **Чистка репозитория:** убрать `.DS_Store`, `temp.html`, `dist/`, вынести `musthavequests.pdf` из репо; дополнить `.gitignore`.
11. **Заменить `alert`/`confirm`** на нативные диалоги/тосты Tauri.
12. Проверить версию `lucide-react` — в `package.json` указана `^1.24.0`, что подозрительно (актуальные версии семейства 0.3xx); убедиться, что это не опечатка/несуществующий тег.

## 6. Предложения по новому функционалу

Исходя из того, что это инструмент координации игровой пачки, наиболее ценные направления:

**Аналитика и история.** Графики динамики казны по дням/неделям, вклад каждого участника за период, история изменений ростера. Данные уже есть — не хватает визуализации (Recharts/Chart.js). Экспорт отчёта дисциплины в CSV/PDF для лидера.

**Автоматизация дисциплины.** Сейчас «сдал ли минимум adena за день» считается вручную по транзакциям. Можно: авто-сброс дневных счётчиков в заданный игровой ресет, история compliance по дням (кто сколько дней подряд выполнял норму), автоматические «предупреждения» по неактивным.

**Уведомления.** Tauri-нотификации в трее: «доступна новая задача prime», «казна пополнена», «ты не сдал дневной минимум». Плагин `tauri-plugin-notification` встаёт поверх текущей архитектуры малой кровью.

**Планировщик активностей / расписание.** Эпические боссы, осады, RB по таймерам с обратным отсчётом и напоминаниями — типовая боль CP в L2, а данные легко хранить в `config`.

**Раздача лута (loot distribution).** Учёт кто что получил из дропа, очередь на предметы, DKP-подобная система баллов — естественное расширение казны.

**Профиль участника.** Персональная страница: класс, уровень, история вкладов, прогресс квестов, статус — сейчас данные размазаны по ростеру и логам.

**Аудит-лог действий** (кто выдал роль, кто отредактировал транзакцию, кто сбросил ростер) — важно и для безопасности (§4.2), и для доверия внутри пачки.

**Быстрые улучшения UX:** глобальный поиск по участникам, тёмная/светлая тема (сейчас, судя по CSS-переменным, только одна), drag & drop перестановка слотов ростера, мобильная/веб-версия (стек это позволяет — фронтенд не зависит от Tauri, кроме updater).

## 7. Резюме

Крепкий, аккуратно написанный проект с хорошей фронтенд-архитектурой и грамотно настроенным периметром Tauri — заметно выше среднего пет-проекта. Главные риски лежат не в коде компонентов, а в **слое данных**: правила Firestore валидируют авторство, но не содержимое; модель прав местами расходится с UX (критичный баг с квестами MEMBER); ряд хранилищ и расчётов не масштабируется. Ближайшие шаги по важности: починить права questLog, добавить валидацию схемы в правила и покрыть их тестами, ввести агрегаты для казны. После этого проект готов к устойчивому росту функционала — от аналитики и уведомлений до loot-distribution.
