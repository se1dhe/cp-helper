# CP-Helper — обзор сервиса, аудит и рекомендации

> Документ подготовлен по результатам адверсиального код-ревью (мульти-агентный прогон: разбор → поиск багов → независимая верификация каждой находки). Проверено ~3200 строк кода. Ниже: сначала **как устроен сервис**, затем **найденные баги** (по убыванию критичности) и **рекомендации** по улучшению.

---

## 1. Что это за сервис

**CP-Helper** — десктопное приложение для управления **CP** (Constant Party / боевой пачкой) в игре Lineage 2, для сквада «0utLaw / UaSqud». Помогает лидеру пачки (Party Leader) вести:

- **ростер** (состав пачки: слоты, классы, уровни, привязка к аккаунтам);
- **казну** (учёт adena и Master Coins — приход/расход);
- **квесты** (чек-листы прокачки по классу/расе для каждого участника);
- **задачи** (общий чек-лист prime/offprime);
- **отчёт по дисциплине** (кто выполнил квесты и сдал минимум adena за день);
- **управление пользователями и ролями**.

Интерфейс на русском и украинском языках.

### Технологический стек

| Слой | Технология |
|------|-----------|
| Оболочка | **Tauri v2** (Rust) — окно, трей, авто-апдейтер, нативные диалоги |
| Фронтенд | **React 19** + **Vite 8**, `react-router-dom` 7 (HashRouter) |
| Бэкенд | **Firebase** — Auth (email/пароль) + **Firestore** (реалтайм-подписки) |
| Иконки/UI | `lucide-react`, кастомные SVG-иконки классов |

Rust-часть почти пустая: окно + трей + плагины `dialog`/`updater`, без собственных команд. **Вся логика живёт во фронтенде и напрямую общается с Firestore.**

## 2. Как это работает

### 2.1. Точка входа и провайдеры

`main.jsx` → `App.jsx`, который:
1. При запуске вызывает Tauri-апдейтер (`check()` → диалог → `downloadAndInstall()`);
2. Оборачивает приложение в `LanguageProvider` → `AuthProvider` → `HashRouter`.

Роутинг (`App.jsx`): один родительский маршрут `/` рендерит `Layout`, а в его `<Outlet/>` — страницы: `Dashboard` (`/`), `Treasury` (`/treasury`), `Roster` (`/roster`), `Admin` (`/admin`), `MemberProgress` (`/members`). **Маршруты не защищены на уровне роутера** — доступ ограничивается только условным рендерингом внутри `Layout` и страниц.

### 2.2. Аутентификация, роли и сессия (`AuthContext.jsx`)

- **Провайдер Email/Password**. Регистрация **открытая** — зарегистрироваться может кто угодно.
- При первом входе клиент **сам создаёт** документ `users/{uid}` в Firestore со `{ email, displayName, nickname, role: "GUEST", createdAt }`.
- Сессия сохраняется штатным механизмом Firebase (`onAuthStateChanged` + локальное хранилище), пользователь остаётся залогинен между запусками.
- **Роли** (поле `role` в `users/{uid}`):
  - `PL` — лидер пачки (полный доступ);
  - `OFFICER` — офицер (казна, задачи, отчёты);
  - `MEMBER` — участник (в основном чтение);
  - `GUEST` — новичок/рекрут (доступа нет, ждёт одобрения).
- Производные флаги: `isPL`, `isOfficer` (= PL ∪ OFFICER), `isGuest`.

`Layout` показывает один из трёх экранов: форму входа/регистрации, экран «ожидайте одобрения» (для GUEST) или полную оболочку с сайдбаром. Ссылки на `/members` и `/admin` видны только PL/OFFICER.

### 2.3. Модель данных Firestore

Всё реалтайм — каждая коллекция/документ имеет `onSnapshot`-подписку.

| Путь | Назначение | Форма |
|------|-----------|-------|
| `users` (коллекция) | пользователи | `{ role, email, nickname, displayName, className, createdAt }` |
| `roster` (коллекция) | слоты пачки (`slot_1`…) | `{ id, className, type, name, lvl, position, userId }` |
| `tasks` (коллекция) | общий чек-лист | `{ text, tag, done, createdAt }` |
| `transactions` (коллекция) | казна (append-only леджер) | `{ type, currency, amount, member, timestamp, addedBy… }` |
| `config/minContributions` | глобальный минимум adena/день | `{ amount }` |
| `config/questData` | датасет квестов (universal + по расам + маппинг класс→раса) | заполняется скриптом `seed_quests.js` |
| `config/questLog` | отметки о выполнении квестов | `{ [slotId]: { [questName]: bool } }` |

**Важные архитектурные особенности:**
- Балансы казны **не хранятся**, а пересчитываются на клиенте суммированием всех транзакций при каждом снапшоте.
- **Нет ни одной транзакции (`runTransaction`)**; единственный `writeBatch` — в `updateRosterNameByUserId`. Остальные операции read-modify-write полагаются на данные, переданные клиентом.
- Отметки квестов ключуются по **`slotId`**, а не по `userId` (см. баг H1).

### 2.4. Страницы

- **Dashboard** — сводка: карточки adena / MC / заполненность пачки, сетка ростера, общий чек-лист задач, сворачиваемая секция квестов по каждому участнику (отметки пишутся отсюда).
- **Roster** — управление составом (только PL): добавить/удалить слот, назначить пользователя, класс, уровень. Назначение пользователя записывает класс в его профиль.
- **Admin** — управление пользователями (PL/OFFICER видят таблицу; только PL меняет роли/ники и сбрасывает ростер).
- **MemberProgress** — отчёт по дисциплине (PL/OFFICER): по каждому участнику — прогресс квестов (`done/total`) и сумма adena за сегодня против минимума → статус «соответствует / не соответствует».
- **Treasury** — банк клана: карточки итогов, курс MC→adena (локальный, не сохраняется), история транзакций; форма добавления записи — только для OFFICER.

### 2.5. Сборка и доставка

- CI (`release.yml`) собирает по тегу `v*` неподписанные сборки: arm64 macOS (dmg) и x64 Windows (nsis). Linux-таргета нет.
- Авто-апдейтер настроен на клиенте (endpoint `se1dhe/cp-helper` + minisign-ключ), **но CI не генерирует подписанный `latest.json`** (см. баг M11).
- Скрипты в корне (`seed_quests.js`, `set_pl.js`, `scrape_icons.js`) — ручные утилиты, не входят в сборку.

## 3. Найденные баги

Обозначения: **C** — critical, **H** — high, **M** — medium, **L** — low. Все находки прошли независимую адверсиальную верификацию (0 отклонено). У части находок критичность была снижена верификаторами, потому что реальный эффект зависит от **правил безопасности Firestore, которых нет в репозитории** (их нельзя проверить по коду).

### 🔴 Критичные

#### C1. Проверка ролей только на клиенте → тривиальная эскалация привилегий до PL/OFFICER
`src/services/adminService.js:13-16`, `src/context/AuthContext.jsx:30-63`

Вся авторизация — это React-условия (`if (!isPL && !isOfficer) return AccessDenied` и т.п.) над полем `role`, которое клиент **сам создаёт и сам меняет**. `updateUserRole` делает `updateDoc(users/{userId}, {role})` без какой-либо проверки прав. Объект `db` уже доступен в области видимости страницы.

**Эксплуатация:** любой залогиненный GUEST может из консоли devtools (или отдельным скриптом с публичным конфигом) выполнить `updateDoc(doc(db,'users',<свойUid>), {role:'PL'})`. После следующего чтения `AuthContext` открывает весь функционал PL: смена ролей других, редактирование ников, сброс ростера, полный контроль казны.

> Реальная эксплуатируемость зависит от развёрнутых правил Firestore. Но в кодовой базе **нет никакой серверной защиты вообще** — а сам факт, что приложение полагается на клиентскую запись роли, указывает на пермиссивные правила.

**Фикс:** в Firestore Rules запретить пользователю менять собственное поле `role`; разрешить смену роли только вызывающему с `role == 'PL'`; при создании `users/{uid}` жёстко фиксировать `role == 'GUEST'`.

### 🟠 Высокие

#### H1. Отметки квестов ключуются по `slotId` → «наследуются» при переназначении слота и сбросе ростера
`src/services/questLogService.js:16`, `adminService.js:34`, `MemberProgress.jsx:71-79`, `Roster.jsx:54-74`

Выполнение квестов пишется в `config/questLog[slotId][questName]`, а не по `userId`. Слоты (`slot_1`…`slot_9`) — стабильные идентификаторы, которые переходят к новым игрокам. При переназначении слота (`Roster.handleSave`) старые отметки **не чистятся**; `initializeRoster` пересоздаёт те же `slot_1..9`, но `config/questLog` не трогает.

**Эффект:** новый участник в слоте наследует выполнение квестов предыдущего. Если класс совпадает, `MemberProgress` покажет его ложно «соответствующим», и офицер примет решение (кик/награда) по искажённым данным. UI-способа очистить осиротевшие отметки нет.

**Фикс:** ключевать по стабильному `userId`, либо чистить записи `questLog` при смене `userId` слота и в `initializeRoster`.

#### H2. `AuthProvider` навсегда застревает на белом экране при любой ошибке Firestore в колбэке авторизации
`src/context/AuthContext.jsx:28-70`

Колбэк `onAuthStateChanged` — `async` с несколькими `await` (`getDoc`, `updateDoc`, `updateProfile`, `setDoc`) **без `try/catch`**. `setLoading(false)` стоит последней строкой. Провайдер рендерит `{!loading && children}`. Если любой `await` упадёт (офлайн без кэша, сетевая ошибка, отказ правил), исключение улетает как unhandled rejection, `loading` навсегда остаётся `true` → **всё приложение — белый экран** без ошибки, спиннера и возможности восстановления, кроме перезагрузки (которая может упасть так же).

**Фикс:** обернуть тело колбэка в `try/catch/finally`, вызывать `setLoading(false)` в `finally`; показать состояние ошибки.

### 🟡 Средние

#### M1. Нет правил безопасности Firestore в репозитории — авторизация на 100% клиентская
`src/firebase.js` — нет `firestore.rules` / `firebase.json`

Публичный веб-конфиг Firebase встроен в клиент (это нормально — ключ не секрет). Но защитить данные могут **только Firestore Rules**, а их нет в системе контроля версий и нет Cloud Functions. Роли читаются чисто клиентом, без custom claims. Если развёрнутые правила окажутся «тестовыми» (`allow if request.auth != null`), любой аккаунт напрямую через SDK читает и переписывает все коллекции. Даже если правила сейчас строгие — они не версионируются и могут молча регрессировать.

**Фикс:** завести версионируемый `firestore.rules` (deny-by-default), деплоить через `firebase.json` в CI; роль — через custom claims либо правила, читающие `role` вызывающего.

#### M2. Открытая саморегистрация
`firebase.js:20-22`, `Layout.jsx:57-59`

Кто угодно из интернета может создать валидный аккаунт и получить `request.auth` identity. Сам по себе — только GUEST без доступа, но это «включатель» для C1 и для любых пермиссивных правил.
**Фикс:** инвайты/allowlist, либо App Check + строгие правила.

#### M3. Полный список email участников читается любым авторизованным пользователем
`adminService.js:5-11`

`subscribeToUsers` тянет всю коллекцию `users` (email, ник, роль) без фильтра, и вызывается не только в Admin, но и на Dashboard. При пермиссивных правилах — утечка PII (таргетированный фишинг).
**Фикс:** ограничить чтение `users` не-гостям; вынести email в отдельный документ, доступный только владельцу и PL.

#### M4. Сохранение слота ростера перезаписывает документ устаревшим снапшотом (lost update)
`Roster.jsx:30-74`

`handleEdit` делает `setEditForm({ ...slot })` — снимок всех полей на момент открытия. Живые обновления форму не рефрешат. `handleSave` пишет **все** поля назад. Если другой PL/офицер поменял слот, пока форма открыта, его изменения молча затираются.
**Фикс:** писать только изменённые поля (diff) или `runTransaction`.

#### M5. Форма казны без блокировки повторной отправки → дубли транзакций
`Treasury.jsx:27-48`

Кнопка submit не блокируется на время `await addTransaction`. Двойной клик / медленная сеть → две записи в леджер. Балансы (сумма всех строк) задваиваются; в `MemberProgress` дублируется дневная adena. Удалить/отредактировать запись через UI нельзя.
**Фикс:** блокировать кнопку на время запроса (как `saving` в Roster) + идемпотентный id.

#### M6. Один пользователь может занимать несколько слотов
`Roster.jsx:54-74`

`handleSave` чистит только прежнего владельца **того же** слота, не проверяя, занят ли назначаемый где-то ещё. В `MemberProgress` участник задваивается, класс перезаписывается по последнему слоту.
**Фикс:** при назначении освобождать другие слоты с тем же `userId` / фильтровать уже назначенных в выпадашке.

#### M7. Первая отметка квеста на пустой БД теряется (рассинхрон `setDoc` vs `updateDoc`)
`questLogService.js:16-24`

`updateDoc` трактует ключ с точкой `${slotId}.${questName}` как **вложенный путь**, а фолбэк `setDoc(..., {merge:true})` (когда документа ещё нет) — как **литеральное имя поля** с точкой. Читатель ищет вложенную структуру и не находит. На свежей БД первая отметка любого квеста теряется после перезагрузки; со второго раза работает.
**Фикс:** в фолбэке строить вложенный объект: `setDoc(ref, { [slotId]: { [questName]: done } }, { merge:true })`.

#### M8. Добавление слота на свежей БД схлопывает 9 дефолтных слотов до 1
`rosterService.js:19-45`

`DEFAULT_ROSTER` (9 слотов) — только клиентский фолбэк для пустой коллекции, в БД он не персистится. `addRosterSlot` пишет один `slot_10` → коллекция перестаёт быть пустой → отображается только один реальный слот. Плюс редактирование дефолтных слотов до персиста падает (`updateDoc` по несуществующему документу).
**Фикс:** персистить `DEFAULT_ROSTER` при первом обращении / в `addRosterSlot`, если коллекция пуста — сначала записать все 9.

#### M9. `parseSortLevel` роняет всё приложение, если `lvl` квеста — не строка
`src/data/quests.js:1-7`

`lvl.match(/^(\d+)/)` внутри компаратора `sort`. Если в `config/questData` `lvl` окажется числом/`null` — `TypeError` во время рендера Dashboard и MemberProgress. **Error boundary в дереве нет** → белый экран всего приложения.
**Фикс:** `String(lvl ?? '').match(...)` + добавить корневой Error Boundary.

#### M10. Квесты с точкой `.` в названии не сохраняют отметку выполнения
`questLogService.js:18`

Тот же механизм field-path, что и в M7: точка в `questName` (напр. «Supplier of Reagents. Part 2») дробит путь при записи, а читатель ищет литеральный ключ. Отметка молча не сохраняется для всех пользователей по этому квесту.
**Фикс:** использовать вложенный объект/FieldPath API или санитизировать/хэшировать ключи квестов.

#### M11. Авто-апдейтер настроен, но CI не производит подписанный манифест — обновления недоставляемы
`.github/workflows/release.yml:39-50`, `tauri.conf.json:50-56`

Три поломки разом: (1) `tauri-action` не получает `TAURI_SIGNING_PRIVATE_KEY` и в конфиге нет `bundle.createUpdaterArtifacts` → `latest.json`/`.sig` не генерируются; (2) `releaseDraft: true`, а endpoint `/releases/latest/download/...` не видит черновики; (3) endpoint указывает на репозиторий `se1dhe/cp-helper` — возможно, не туда, куда публикуются сборки. Состояние fail-safe (не MITM), но канал обновлений не работает — патчи не доедут до пользователей.
**Фикс:** добавить секреты подписи в CI, `createUpdaterArtifacts: true`, публиковать не-черновые релизы, сверить endpoint; CI-проверка, что `latest.json` валиден против pubkey.

### 🟢 Низкие

- **L1.** `initializeRoster` — неатомарный цикл delete-then-recreate; при обрыве оставляет частичное состояние (лечится повторным сбросом, т.к. фиксированные id). `Admin.jsx` без `try/catch` — нет обратной связи об ошибке. `adminService.js:34-43`
- **L2.** Казна ключуется по строке-имени участника; путь переименования в Admin (`updateUserNickname`) расходится с путём самопереименования в Layout (обновляет и ростер). Возможны рассинхрон отображения и «осиротевшие» контрибуции при самопереименовании. `MemberProgress.jsx:61-69`
- **L3.** `handleSave` делает до трёх независимых записей (слот + класс нового + очистка старого) без батча — при сбое частичный рассинхрон класса. `Roster.jsx:54-74`
- **L4.** Один битый документ транзакции (не число в `amount`) отравляет итог казны значением `NaN` (в ru-RU рендерится как «не число»). Через штатную форму невозможно (там `Number()`), только импорт/ручная правка. `treasuryService.js:29-33`
- **L5.** `ClassIcon` не сбрасывает флаг `useSvg` при смене пропа `className` — после разовой ошибки загрузки иконка залипает на SVG-фолбэке. Косметика. `ClassIcon.jsx:41-65`
- **L6.** Скрипты `set_pl.js` и `seed_quests.js` пишут в прод-Firestore **без аутентификации** — сработают только при открытых правилах; если правила строгие, скрипты просто упадут. Стоит перевести на Admin SDK с сервис-аккаунтом. `set_pl.js`, `seed_quests.js`
- **L7.** Нет CI-проверки соответствия версий `package.json` ↔ `tauri.conf.json` ↔ git-тег (важна версия из `tauri.conf.json` — по ней сравнивает апдейтер). `package.json:4`, `tauri.conf.json:4`
- **L8.** `scrape_icons.js` пишет в `public/` байты со стороннего сайта; тяжёлая зависимость `puppeteer` ради разового скрейпа; мелкий баг приоритета операторов в фильтре. `scrape_icons.js:42-74`

## 4. Рекомендации (что добавить и улучшить)

### 4.1. Безопасность — приоритет №1

1. **Завести `firestore.rules` в репозиторий** (deny-by-default) и деплоить из CI через `firebase.json`. Это закрывает C1, M1, M2, M3, L6 разом. Ключевые правила:
   - поле `role` неизменяемо владельцем; менять может только `PL`;
   - при создании `users/{uid}` — `role == 'GUEST'` и `uid == request.auth.uid`;
   - чтение `users` — только не-гостям; email вынести в приватный подрезурс;
   - записи в `roster`, `transactions`, `config/*` — по конкретным ролям.
2. **Перевести роли на Firebase custom claims** (через Cloud Function/Admin SDK), а не на читаемое клиентом поле.
3. **Включить [Firebase App Check](https://firebase.google.com/docs/app-check)**, чтобы конфиг работал только из настоящего бинарника.
4. **Закрыть или заинвайтить регистрацию** (allowlist/инвайты).
5. Утилиты `set_pl.js`/`seed_quests.js` перевести на **Admin SDK** с сервис-аккаунтом.
